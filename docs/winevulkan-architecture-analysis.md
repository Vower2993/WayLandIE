## Critical Architecture: Two-Process Model

Wine's Vulkan implementation splits across TWO address spaces:

### PE Side (winevulkan.dll — runs in Windows process context)
- Loaded into the Windows process as `winevulkan.dll`
- Exports standard Vulkan API functions (`vkCreateInstance`, `vkCreateDevice`, etc.)
- Each function is a thin wrapper that marshals parameters and calls the Unix side via `UNIX_CALL()`
- The PE-side `VkDevice` handle is `struct VkDevice_T*`

### Unix Side (winevulkan.so — runs in Wine's Unix process)
- Contains the actual Vulkan dispatch logic
- Calls the host system's Vulkan driver (libvulkan.so on Linux, adrenotools on Android)
- The Unix-side device object is `struct vulkan_device`

## Handle Indirection Chain

A Windows `VkDevice` handle goes through TWO levels of indirection:

```
VkDevice (Windows pointer)
  → struct VkDevice_T* (PE-side struct)
      → obj.loader_magic = 0x01CDC0DE  (offset 0, 8 bytes)
      → obj.unix_handle  = pointer to struct vulkan_device  (offset 8, 8 bytes)
      → extensions       (offset 16, bitfield struct)
      → queues[1]        (variable-length array)
```

### struct vulkan_client_object (offset 0 of every dispatchable handle)
```c
struct vulkan_client_object {
    UINT64 loader_magic;   // 0x01CDC0DE (NOT 0x10aded040410aded — that's the loader's magic)
    UINT64 unix_handle;    // pointer to the Unix-side object (struct vulkan_device, etc.)
};
```

### struct VkDevice_T (PE side — what DXVK sees as VkDevice)
```c
struct VkDevice_T {
    struct vulkan_client_object obj;  // offset 0: loader_magic + unix_handle
    struct vulkan_device_extensions extensions;  // offset 16: bitfield of enabled extensions
    struct VkQueue_T queues[1];       // variable-length array of queue handles
};
```

### struct vulkan_device (Unix side — contains the REAL function pointers)
```c
struct vulkan_device {
    VULKAN_OBJECT_HEADER(VkDevice, device);  // host.device + client pointer
    struct vulkan_physical_device *physical_device;
    struct vulkan_device_extensions extensions;
    
    // ALL_VK_DEVICE_FUNCS expands to one field per Vulkan device function:
    PFN_vkAcquireNextImage2KHR p_vkAcquireNextImage2KHR;
    PFN_vkAcquireNextImageKHR p_vkAcquireNextImageKHR;
    PFN_vkAllocateMemory p_vkAllocateMemory;
    PFN_vkCreateBuffer p_vkCreateBuffer;
    PFN_vkCreateSwapchainKHR p_vkCreateSwapchainKHR;
    PFN_vkDestroySwapchainKHR p_vkDestroySwapchainKHR;
    // ... ~300+ named function pointer fields ...
    PFN_vkQueuePresentKHR p_vkQueuePresentKHR;
    // ...
    
    uint64_t queue_count;
    struct vulkan_queue *queues;
    VkQueueFamilyProperties *queue_props;
};
```

**THE p_vkXxx FIELDS ARE ON THE UNIX SIDE, NOT THE PE SIDE.**

The PE-side `struct VkDevice_T` does NOT contain function pointers. It only has `loader_magic`, `unix_handle`, `extensions`, and `queues[]`. The function pointers are in `struct vulkan_device` on the Unix side, accessed via `obj.unix_handle`.

## How Dispatch Actually Works

### PE-side vkGetDeviceProcAddr (loader.c:135)
```c
PFN_vkVoidFunction WINAPI vkGetDeviceProcAddr(VkDevice device, const char *name) {
    // ... extension checks ...
    if (is_available_device_function(device, name)) {
        func = wine_vk_get_device_proc_addr(name);  // name-based linear search
        if (func) return func;
    }
    // ...
}
```

`wine_vk_get_device_proc_addr` does a LINEAR SEARCH through `vk_device_dispatch_table[]` — a static array of `{name, func_ptr}` pairs. It does NOT access any per-device dispatch table.

### What wine_vk_get_device_proc_addr returns
It returns PE-side thunk functions like `vkCreateSwapchainKHR` (defined in the PE dll). These thunks:
1. Marshal Windows structs → Unix structs (handle 32/64-bit pointer translation)
2. Call `UNIX_CALL(vkCreateSwapchainKHR, &params)` — crosses PE→Unix boundary
3. The Unix side unwraps the device handle and calls `device->p_vkCreateSwapchainKHR(host_device, ...)`

### Unix-side call chain (vulkan.c)
```c
// Unix-side thunk (generated by make_vulkan)
NTSTATUS vkCreateSwapchainKHR(void *args) {
    struct vkCreateSwapchainKHR_params *params = args;
    VkDevice client_device = params->device;
    struct vulkan_device *device = vulkan_device_from_handle(client_device);
    // vulkan_device_from_handle does:
    //   client = (struct vulkan_client_object *)handle;
    //   return (struct vulkan_device *)(UINT_PTR)client->unix_handle;
    
    // Call the host driver via the stored function pointer:
    VkSwapchainKHR host_swapchain;
    VkResult res = device->p_vkCreateSwapchainKHR(device->host.device, &create_info_host, NULL, &host_swapchain);
    // ... wrap the result ...
}
```

## The Dispatch Table We've Been Trying to Patch — DOESN'T EXIST

**Our entire approach has been based on a false premise.** We assumed winevulkan has a flat function-pointer array at offset 8 of `VkDevice` that we could patch. The reality:

1. **Offset 0 of VkDevice**: `loader_magic` (0x01CDC0DE) — NOT a pointer to a dispatch table
2. **Offset 8 of VkDevice**: `unix_handle` — a pointer to `struct vulkan_device` on the Unix side
3. **The function pointers** (`p_vkXxx`) are in `struct vulkan_device`, which lives in the **Unix process** address space — NOT accessible from the PE side where our Vulkan layer runs

When we read `(void**)((char*)device + 8)` and found ~200 non-null entries, we were reading `struct vulkan_device`'s fields starting from its beginning:
- `VULKAN_OBJECT_HEADER` (host.device pointer + client pointer) — 16 bytes
- `physical_device` pointer — 8 bytes  
- `extensions` bitfield struct — ~16 bytes
- Then the `p_vkXxx` function pointers begin

The "indices 253-257" we found were just coincidental offsets within the `struct vulkan_device` where swapchain function pointers happen to land. These offsets are determined by:
- The number of `p_vkXxx` fields that come before `p_vkCreateSwapchainKHR` in the struct
- Which is determined by `ALL_VK_DEVICE_FUNCS` in make_vulkan, which iterates `self.registry.device_funcs` in XML order

**These indices WILL change if:**
- The Vulkan registry (vk.xml) adds new functions before the swapchain ones
- Wine rebuilds with a different Vulkan version
- Extensions are enabled/disabled (no — the struct has ALL functions regardless of enabled extensions)

## Why Our Approach Cannot Work

Our Vulkan layer (`libvk_layer_waylandie_dmabuf.so`) is loaded by the **PE-side** Vulkan loader. It intercepts `vkGetInstanceProcAddr`/`vkGetDeviceProcAddr` calls. But:

1. **The PE-side thunks are what DXVK calls.** When DXVK calls `vkCreateSwapchainKHR`, it calls the PE-side thunk in `winevulkan.dll`, which marshals and calls the Unix side.

2. **Patching `struct vulkan_device`'s function pointers would only affect the Unix side.** But our layer runs in the PE process. The Unix side is a separate shared memory space accessed via `UNIX_CALL()`.

3. **Even if we could patch `struct vulkan_device`, the function pointers are host driver functions** (e.g., the real Turnip/Adreno `vkCreateSwapchainKHR`). Replacing them with our hooks would require our hooks to run in the Unix process context.

## What Actually Needs to Happen

To intercept Vulkan swapchain calls for dmabuf forwarding, we have three viable approaches:

### Approach A: Patch the PE-side dispatch table (NAME-BASED)
The PE side has `vk_device_dispatch_table[]` — a static array of `{name, func_ptr}` pairs. `wine_vk_get_device_proc_addr` does a linear search by name. We could:
1. Find this table in winevulkan.dll's memory
2. Replace the `func_ptr` for `vkCreateSwapchainKHR` etc. with our hooks
3. Our hooks call the original thunk, then do the dmabuf forwarding

**Problem:** The table is `static const` — in read-only memory. We'd need to mprotect it. Also, the original thunk calls the Unix side, which calls the host driver — we can't intercept the dmabuf there.

### Approach B: Hook at the Unix side (winevulkan.so)
Patch `struct vulkan_device`'s `p_vkXxx` function pointers. These ARE writable and ARE the actual host driver function pointers. We'd need to:
1. Find the Unix-side `struct vulkan_device` from the PE-side `VkDevice` handle (read `obj.unix_handle` at offset 8)
2. Cross the PE→Unix boundary (the Unix object is in a different address space in some Wine configurations, but in the WinNative/Android setup they share the same process)
3. Patch the `p_vkCreateSwapchainKHR` etc. fields with our hooks
4. Our hooks run in Unix context and can call `vkGetMemoryFdKHR` etc.

**This is actually what we've been trying to do** — but we were reading the wrong offset. The `unix_handle` at offset 8 gives us the `struct vulkan_device*`. We need to FOLLOW that pointer, then offset into the struct to find the `p_vkXxx` fields.

### Approach C: Build winevulkan from source with our hooks compiled in
Modify `make_vulkan` to generate thunks that call our dmabuf forwarding code. This is the cleanest but requires rebuilding winevulkan.so.

## The Correct Pointer Chain for Approach B

```
VkDevice device (PE handle)
  → (struct vulkan_client_object *)device
      → .unix_handle  [offset 8, 8 bytes] → struct vulkan_device*
          → .host.device  [offset 0-7, the host VkDevice]
          → .physical_device  [offset 8-15]
          → .extensions  [offset 16-31, bitfield]
          → .p_vkXxx  [offset 32+, function pointers]
              The exact offset of p_vkCreateSwapchainKHR depends on
              how many p_vkXxx fields precede it in ALL_VK_DEVICE_FUNCS.
```

To find the correct offset of `p_vkCreateSwapchainKHR` within `struct vulkan_device`, we need to either:
1. Build winevulkan from source and use `offsetof(struct vulkan_device, p_vkCreateSwapchainKHR)`
2. Count the fields in `ALL_VK_DEVICE_FUNCS` that come before `vkCreateSwapchainKHR`
3. Use `dlsym` on the Unix side to get the address of a known function, then scan the struct

## The `is_available_device_function` Check (loader.c:173)
```c
if (is_available_device_function(device, name)) {
    func = wine_vk_get_device_proc_addr(name);
    if (func) return func;
}
```
`is_available_device_function` calls the Unix side to check if a function is available. If our layer's `vkGetDeviceProcAddr` is called during `vkCreateDevice` (by winevulkan's internal init), and we call `fp_gdpa` which triggers this check, it can cause re-entrancy and assertions.

## Summary of What We've Been Doing Wrong

1. **Wrong struct**: We patched `(void**)((char*)device + 8)` thinking it was a flat dispatch table. It's actually `unix_handle` — a pointer to `struct vulkan_device`. Reading it as `void**` gives us the FIELDS of `struct vulkan_device`, not a function pointer array.

2. **Wrong process**: Our layer runs in the PE process. The function pointers we need to patch are in `struct vulkan_device` on the Unix side. In the WinNative/Android setup, these share the same process, so the pointer IS accessible — but we need to FOLLOW it.

3. **Wrong indices**: The "indices 253-257" were coincidental offsets within `struct vulkan_device` where swapchain function pointers happen to land. They're not stable — they depend on the exact set of `ALL_VK_DEVICE_FUNCS` generated by `make_vulkan`.

## What to Do Next

The correct approach is:
1. Read `unix_handle` from `(uint64_t*)((char*)device + 8)` — this gives us `struct vulkan_device*`
2. Follow that pointer to the Unix-side struct
3. Find the `p_vkCreateSwapchainKHR` field offset — either by building from source with `offsetof`, or by scanning the struct for known function pointer values (resolved via `vkGetDeviceProcAddr` on the Unix side)
4. Patch those fields with our hooks

The hooks would then run in the Unix process context and have direct access to the host Vulkan driver.

## The Real Problem with Our Approach

Our Vulkan layer runs in the **PE process**. When we read `(void**)((char*)device + 8)`, we get `unix_handle` — a pointer to `struct vulkan_device` in the **Unix address space**. In Wine's architecture, PE and Unix share the same process, so we CAN follow this pointer.

But the function pointers in `struct vulkan_device` are **HOST driver functions** (Turnip/Adreno), not winevulkan thunks. We can't match them by value because we don't have access to the host driver's function addresses from the PE side.

## Three Viable Approaches (Ranked by Reliability)

### 1. Build winevulkan from source with hooks (MOST RELIABLE)
Modify `make_vulkan` to insert dmabuf forwarding code into the generated thunks for `vkCreateSwapchainKHR`, `vkQueuePresentKHR`, etc. The CI already builds winevulkan from source (`.github/scripts/build-winewayland-driver.sh`), so this is feasible.

**Pros:** Exact offsets known at compile time, no runtime patching needed, no mprotect needed.
**Cons:** Requires modifying the winevulkan build, larger code change.

### 2. Patch struct vulkan_device by scanning for known host function pointers
1. Read `unix_handle` from `VkDevice_T.obj.unix_handle` (offset 8)
2. Follow to `struct vulkan_device`
3. Call the Unix-side `vkGetDeviceProcAddr` (via `fp_gdpa`) to get host function pointers
4. Scan the `p_vkXxx` array for matching pointers
5. Replace with our hooks

**Pros:** No build modification needed, works at runtime.
**Cons:** Requires calling `fp_gdpa` which triggers assertions during `vkCreateDevice`. Must defer to after device creation completes. Also, `fp_gdpa` returns PE-side thunks, not host functions — won't match.

### 3. Patch the PE-side static dispatch table
The PE-side `winevulkan.dll` has `vk_device_dispatch_table[]` — a static const array of `{name, func_ptr}`. We can find it in memory and replace the `func_ptr` for swapchain functions with our hooks.

**Pros:** Our hooks would intercept at the PE thunk level, before Unix-side dispatch.
**Cons:** The table is `static const` (in `.rodata`), needs mprotect. Also, the PE thunks marshal structs before calling Unix side — our hooks would need to handle the marshaled format.

## Recommendation

**Approach 1 (build from source)** is the most reliable and is the only approach that can work with 100% certainty. The CI already builds winevulkan from source — we just need to add a patch that inserts our hook code into the generated thunks.

The patch would modify `make_vulkan` to generate thunks that, for swapchain/present functions, call our dmabuf forwarding library before/after calling the host driver. This eliminates ALL the runtime patching complexity.
